## 2. 마일스톤 현실성 평가

### 총 4.5주 → ⚠️ 타이트하지만 가능 (조건부)

| 단계 | 평가 | 코멘트 |
|------|------|--------|
| M1 (1주) | ⚠️ 약간 타이트 | RN 프로젝트 셋업 + STT 연동은 디바이스 권한/스트리밍 처리로 1주 빠듯. **STT는 Google 대신 Whisper로 갈 경우 서버사이드 처리라 더 간단할 수 있음** |
| M2 (1주) | ✅ 적정 | FastAPI + OpenAI 연동은 1주면 충분 |
| M3 (1.5주) | ⚠️ 핵심 리스크 | TTS + 발음 평가가 **외부 API 3개**(ElevenLabs, Azure Speech, Google STT) 연동. API 키 셋업, 오디오 포맷 변환, 에러 처리 등 예상보다 시간 소요 가능 |
| M4 (0.5주) | ❌ 부족 | 히스토리 + UI 다듬기 0.5주는 비현실적. UI 다듬기만 1주는 잡아야 함 |
| M5 (0.5주) | ⚠️ 조건부 | 개인 서버 배포라면 가능하지만 앱스토어 배포 포함 시 최소 1~2주 추가 |

### 제안: 총 6~7주로 조정

```
M1: RN 셋업 + 홈/일기 작성 + STT         — 1.5주
M2: 백엔드 API + OpenAI 연동              — 1주
M3: 학습 화면 + TTS + 발음 평가           — 2주
M4: 히스토리 + UI 폴리싱 + 버그 수정      — 1주
M5: 테스트 + 배포                         — 1주
합계: ~6.5주
```

## 3. 우선순위 조정 제안

### API 설계 수정

현재 `POST /api/diary`와 `POST /api/diary/{id}/translate`가 분리되어 있는데, 사용자 플로우에서는 "번역하기" 버튼 한 번에 저장+번역이 동시에 일어남. **두 가지 옵션:**

- **A)** `POST /api/diary`에서 저장과 번역을 한 번에 처리 (UX 단순화, 현재 플로우와 일치)
- **B)** 분리 유지하되 프론트에서 연속 호출 (번역 재시도 가능)

→ **B안 권장.** 번역 실패 시 재시도가 가능하고, 초안 저장 기능도 자연스러움.

### STT 기술 선택 재검토

PRD에는 Google Speech-to-Text, 리스크 대응에는 Whisper API 언급. **통일 필요.**
- Whisper API: 서버사이드, 높은 한국어 정확도, 실시간 스트리밍 불가
- Google STT: 클라이언트 스트리밍 가능, 실시간 텍스트 표시

→ 실시간 피드백이 UX에 중요하면 Google STT, 정확도 우선이면 Whisper. **결정 후 PRD 통일.**

### TTS 기술

PRD 본문은 ElevenLabs, 리스크 대응은 OpenAI TTS 언급. 이것도 **통일 필요.**

### Phase 2 번호 오류

F10이 "레벨 테스트"와 "통계 대시보드" 두 곳에 중복 사용됨. F7도 빠져있음.

## 4. 백엔드/프론트엔드 팀 전달 사항

### 백엔드 (FastAPI)

1. **인증 미들웨어** 설계 필요 — JWT 또는 세션 기반. users 테이블에 email/password 또는 OAuth provider 컬럼 추가
2. **OpenAI 프롬프트 설계** 문서화 — 학습 포인트 추출 프롬프트의 JSON 스키마를 명확히 정의 (card_type, cefr_level 등 enum 값)
3. **오디오 파일 저장 전략** — pronunciation_results.audio_url이 있는데, S3/로컬 스토리지 정책 미정. 개인 서버라면 디스크 저장 + 정리 배치 필요
4. **API 응답 스키마** — PRD에 JSON 응답 형태 명세 없음. OpenAPI/Swagger 스펙 작성 필요
5. **일기 길이 제한** — max_length 정책 결정 후 validation 적용
6. **DB 추가 고려사항:**
   - `diaries` 테이블에 `updated_at` 컬럼 추가 (수정 기능 대비)
   - `users` 테이블에 `email`, `auth_provider` 등 인증 관련 컬럼 추가
   - `learning_cards`에 `card_type = 'full'`의 용도가 불명확 — 번역문 전체를 카드로 쓰는 건지 정의 필요

### 프론트엔드 (React Native)

1. **오디오 권한 처리** — iOS/Android 마이크 권한 요청 플로우, 권한 거부 시 대체 UX
2. **오프라인 대응** — MVP에서도 네트워크 끊김 시 로딩/에러 상태 UI 필요
3. **STT 실시간 vs 배치** — 기술 선택에 따라 UI가 달라짐 (실시간 텍스트 vs 녹음 후 변환)
4. **학습 카드 UX** — 카드 스와이프? 버튼? 제스처 정의 필요
5. **다크모드** — 일기 앱 특성상 밤에 쓰는 유저 많음. MVP부터 고려 권장
6. **키보드 입력 지원** — STT 외 텍스트 직접 입력 UI 반드시 포함

## 5. DB 모델 리뷰

### ✅ 잘된 점
- ERD가 깔끔하고 MVP에 적합한 최소 구조
- CASCADE 삭제, 인덱스 설계 적절
- CEFR 레벨 체계 반영

### ⚠️ 개선 사항

| 항목 | 현재 | 제안 |
|------|------|------|
| `users` 인증 | nickname만 있음 | email, password_hash 또는 auth_provider 추가 |
| `diaries.updated_at` | 없음 | 수정 기능 대비 추가 |
| `learning_cards.card_type` | word/phrase/full | 'full'의 정의 명확화 필요. '번역문 전체 따라 말하기'인지? |
| 복습 데이터 | 없음 | Phase 2 대비 `review_history` 테이블 예비 설계 |
| 소프트 삭제 | 없음 | `diaries`에 `deleted_at` 컬럼 추가 권장 (실수 복구) |
| TTS 캐시 | 없음 | 같은 텍스트 반복 TTS 호출 방지를 위한 캐시 테이블 또는 파일 캐시 전략 |

## 6. 종합 의견

전체적으로 **MVP 범위가 잘 정의**되어 있고, 사용자 플로우가 상세하게 그려져 있어 개발팀이 이해하기 좋은 PRD입니다.

**가장 시급한 3가지:**
1. 🔴 **인증 플로우 추가** — 가장 큰 누락. 디바이스 기반 익명 계정이라도 정의 필요
2. 🔴 **기술 스택 통일** — STT(Google vs Whisper), TTS(ElevenLabs vs OpenAI) 결정
3. 🟡 **마일스톤 현실화** — 4.5주 → 6.5주로 여유 확보

이 3가지만 보완하면 바로 개발 착수할 수 있는 수준입니다.
